## Delivery cadence — one feature at a time

Work proceeds strictly feature-by-feature down the §5 Progress checklist. No batching, no parallel waves. Per [ADR 0014](adr/0014-agents-as-default-entry-point.md), the default flow is agent-led:

1. [`@idea-intake`](../.github/agents/idea-intake.agent.md) — classify the work, enforce the §6 dependency-matrix and §8 ADR gates inline when the work maps onto a Progress-checklist row, draft the issue, create and push the scoped feature branch on confirmation.
2. [`/build-item`](../.github/prompts/build-item.prompt.md) — implement, locally validate, deploy to the lab, verify exit criteria. Loops until the author replies `ready for checkin`. **Never commits or pushes.**
3. [`@artifact-resolver`](../.github/agents/artifact-resolver.agent.md) — run `/build-item` validation, stage scoped paths, run the secrets-scan, commit with Conventional Commits, push, open the PR with all required sections.
4. [`@owner-approval`](../.github/agents/owner-approval.agent.md) — apply `owner-approved`, wait for the [`pr-auto-merge.yml`](../.github/workflows/pr-auto-merge.yml) workflow to squash-merge, run local cleanup, confirm the §5 checkbox tick, prompt for the next feature.

[`@squad`](../.github/agents/squad.agent.md) is reserved for content-creation interviews ([`/add-classification`](../.github/prompts/add-classification.prompt.md), [`/add-data-source`](../.github/prompts/add-data-source.prompt.md)) and persona-led discussion. It is not a lifecycle agent.

Rules that make the cadence work:

- **One feature per PR.** If a diff starts growing beyond the scoped feature, stop and split.
- **Dependencies are hard gates.** A feature row cannot start until every prerequisite in §6 is ticked. `@idea-intake` Step 0 enforces this.
- **Re-runnable.** Every deploy must remain idempotent. v2 never lowers the v1 idempotency bar.
- **No skipping.** Triviality is not an excuse to batch.
- **ADRs block the first row they gate, not the whole feature group.** An open §8 question blocks only the rows it names.

## 1. Scope and constraints

- **Target environment.** Single lab environment: `purview-contoso-lab` in `rg-purview-lab`, region `eastus`, tenant `contoso.onmicrosoft.com`. Any multi-environment story is a future design PR, not part of this plan. See the "Environment and identifier boundaries" section of [`.github/copilot-instructions.md`](../.github/copilot-instructions.md).
- **Two planes stay separate.** Control plane stays in [`infra/`](../infra/); data plane stays in [`data-plane/`](../data-plane/) + [`scripts/`](../scripts/). Any new top-level folder requires an ADR.
- **Licensing assumption.** M365 E5 or E5 Compliance add-on across the tenant. Verified by [`scripts/Test-M365Licensing.ps1`](../scripts/Test-M365Licensing.ps1) per [ADR 0001](adr/0001-m365-licensing-verification.md).
- **Automation identity.** App-only auth via certificate-in-Key-Vault and federated OIDC from GitHub Actions, per [`.github/instructions/security.instructions.md`](../.github/instructions/security.instructions.md) rules #2 and #3. No stored client secrets.
- **v1 evidence stands.** v2 reviews start from the current tenant state. v1 deployments are not re-litigated; if a v2 review surfaces drift, drift is closed by a new v2 item (not by amending a v1 PR).
- **No real data, no real identifiers.** Synthetic substitutes only, per [`.github/instructions/sample-data.instructions.md`](../.github/instructions/sample-data.instructions.md).

## 2. Guiding principles

1. **One Purview feature end-to-end before the next.** v1 used dependency-tiered waves to bootstrap the foundation. v2 uses per-feature lanes because the foundation is stable: each feature row goes review → close drift → harden as-code → tick, fully, before the next row begins.
2. **Microsoft Learn is authoritative.** Every feature claim, capability assertion, role name, cmdlet, REST endpoint, and resource type cites a current Learn page in the same PR that introduces or modifies it. No training-data-only claims. See [`.github/copilot-instructions.md`](../.github/copilot-instructions.md) "Grounding" section.
3. **Drift is closed, not papered over.** A v2 review that finds the live tenant out of step with the desired-state YAML closes the gap one of three ways: (a) update the YAML to match intended state and re-apply, (b) ratify the deviation via a new ADR, or (c) open a follow-up issue with an explicit owner and a target date. "We'll fix it later" without one of those three artifacts is not acceptable.
4. **Simulation-first stays.** Every state-changing reconciler keeps its `-WhatIf` mode. v2 raises the bar by requiring `-WhatIf` to be re-run against the live tenant during each feature review and pasted into the PR description.
5. **Idempotent and reversible.** Destructive flags (`-PruneMissing`, `-Force`) are opt-in and carry the `destructive` PR label, unchanged from v1.
6. **Watch-list re-opening.** A row sitting on a v1 watch-list (no documented authoring surface from Microsoft) is promoted into a real Progress-checklist row the moment Microsoft publishes a documented Graph / REST / PowerShell authoring surface. The promotion event must cite the Learn page that documents the new surface.

## 4. Per-feature lifecycle (review → close drift → harden → tick)

Each §5 row is one `@idea-intake` item. The item runs through four phases before the box can be ticked:

1. **Review current state.** Capture the live tenant's configuration for this feature (PowerShell `Get-*`, REST `GET`, or portal export). Diff against the desired-state YAML. Record findings in the issue body.
2. **Close drift.** For each diff entry, pick one of: (a) update the YAML and re-apply, (b) ratify the deviation via a new ADR, or (c) file a follow-up issue with an owner and a target date. No silent acceptance.
3. **Harden as-code.** Add the gaps the review surfaces: missing Pester coverage, missing `-WhatIf` evidence in the workflow, missing solution-doc page under [`docs/solutions/`](solutions/), missing schema, missing Learn citation. Cap with an end-to-end re-deploy from a clean checkout.
4. **Tick the §5 row.** The `@owner-approval` agent ticks the row only after the PR includes (a) the review diff, (b) the drift-closure evidence, (c) the hardening artifacts, and (d) a fresh `-WhatIf` against the live tenant.

A feature whose v1 row is a watch-list entry (Communication Compliance, DSPM for AI) follows a slightly different path: phase 1 confirms whether Microsoft has published a programmatic authoring surface since v1 closed. If yes, the row is promoted and the lifecycle continues. If no, the v2 review consists of re-verifying the read-only posture and re-confirming the watch-list status; the box is ticked on that basis alone, citing the Learn pages confirming "no programmatic authoring surface" as of the review date. The reconciler for such a row is **drift-detection only** for the duration of the watch-list state: it stays in place to surface portal-authored objects in `-WhatIf` reports, but the desired-state YAML ships an empty collection and never authors. Communication Compliance currently sits in this posture per [ADR 0019](adr/0019-cc-graph-pivot.md).

- Porting the reference repo's Azure DevOps pipelines verbatim. This repo stays on GitHub Actions — same as v1.

- eDiscovery-as-code (cases, custodians, holds, searches, review sets, exports) — same as v1, per [ADR 0018](adr/0018-ediscovery-scope.md).
