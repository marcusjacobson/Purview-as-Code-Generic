# 0033 — DLP rule tracked-field expansion: ratify the per-field shape contract, amend ADR 0033 in place, ship one PR per 1-3 fields, no per-field ADR

- **Status:** Accepted
- **Date:** 2026-06-04
- **Gates:** Resolves the design question raised by umbrella issue [#521](../../issues/521). Unblocks per-field implementation PRs filed as children of [#521](../../issues/521). Does not appear in [`docs/project-plan.md`](../project-plan.md) §8 Open-question ADRs — [#521](../../issues/521) is cross-cutting expansion of an existing reconciler, not a wave-blocking question.
- **Deciders:** @contoso

## Context

The DLP rule schema currently in [`data-plane/dlp/policies.schema.json`](../../data-plane/dlp/policies.schema.json) models **10 of the ~190 writable parameters** that `New-DlpComplianceRule` accepts:

- `name`, `priority`
- `sensitiveInfoTypes`, `sensitivityLabels`, `advancedRule` (predicate; see [ADR 0031](0031-dlp-advancedrule-yaml-shape.md))
- `blockAccess`, `notifyUser`, `generateIncidentReport`, `generateAlert`
- `notes` (pass-through marker introduced by PR #516)

That shape was set during the initial DLP drift closure ([#362](../../issues/362), PR #383) and extended twice for predicate-shape concerns ([ADR 0031](0031-dlp-advancedrule-yaml-shape.md), [ADR 0032](0032-dlp-generic-locations-shape.md)). It did **not** establish a parent shape ADR for the rule surface as a whole — unlike [ADR 0015](0015-label-policy-shape.md) for label policies, which 0030 amends in place. This ADR is the missing parent.

The live-tenant cross-check during PR for [#515](../../issues/515) and re-confirmed by the [`.copilot-tracking/dlp-rule-field-probe.ps1`](../../.copilot-tracking/dlp-rule-field-probe.ps1) probe (2026-06-04, 16 lab-tenant rules) shows the following operator-meaningful rule fields that the schema does **not** model:

**Populated on every lab-tenant rule (16/16), Microsoft-shipped default values (5):**

| Field | Type | Sample value | Source-of-decision |
|---|---|---|---|
| `EnforcePortalAccess` | bool | `True` | Microsoft default; portal toggle |
| `NotifyEmailExchangeIncludeAttachment` | bool | `True` | Microsoft default; portal toggle |
| `ReportSeverityLevel` | enum (Low/Medium/High) | `Low` | Microsoft default; portal selector |
| `Comment` | string | `"This rule is matched if 1 to 9 credit card numbers..."` | Microsoft template; operator-editable |
| `AccessScope` | enum (InOrganization / NotInOrganization / PerUser) | `NotInOrganization` | Microsoft template (set on 2/16 credit-card rules); portal selector |

The first three are 16/16 with the same Microsoft default value across every rule; their **drift surface** is empty in the lab today but a single portal flip on any one of them would silently miss detection. The last two carry operator-meaningful string / enum content that already varies across rules.

**Populated on some lab-tenant rules (1-3 of 16), operator-meaningful (9):**

| Field | Type | Rules | Decision |
|---|---|---|---|
| `EndpointDlpRestrictions` | object[] | 2 (endpoint policies) | Structured; per-app enforcement matrix |
| `AlertProperties` | hashtable | 1 (Copilot policy) | Structured; per-channel alert routing |
| `RestrictAccess` | string[] | 1 (Copilot policy) | List of restricted action identifiers |
| `NotifyEmailCustomText` | string | 2 | Operator string |
| `NotifyEmailOnedriveRemediationActions` | enum | 3 | OneDrive remediation pick-list |
| `NotifyOverrideRequirements` | enum | 3 | Override-requirement pick-list |
| `NotifyPolicyTipCustomText` | string | 3 | Operator string |
| `NotifyPolicyTipDisplayOption` | enum | 3 | Tip / NotifyOnly / NotificationOnly |
| `NotifyUserType` | enum | 3 | Recipient role enum |
| `NotifyAllowOverride` | enum | 1 | `WithJustification` etc. |
| `IncidentReportContent` | enum[] | 1 | Field-set for incident-report payload |

Plus ~30 system / computed fields the probe surfaced (`CreatedBy`, `DistinguishedName`, `ExchangeVersion`, `Id`, `ObjectClass`, `ObjectState`, `OrganizationalUnitRoot`, `WhenCreatedUTC`, `WhenChangedUTC`, `ProcessingLimitExceeded`, the 8 `*Is*Not*` / `*HasSender*` booleans uniformly `False`, etc.) that ADR 0029 §"Excluded from tracking" already classifies as tenant-only state. Those stay out of scope here, same as ADR 0015 §"Tenant-only state" did for label policies.

[#521](../../issues/521) names 14 fields and explicitly asks for a shape contract before any per-field PR ships. The [`@idea-intake`](../../.github/agents/idea-intake.agent.md) for the first child PR is gated on this ADR existing at `Status: Accepted` — same gate the seven label-policy children faced for [ADR 0030](0030-label-policies-tracked-field-expansion.md).

Three candidate shapes were on the table, mirroring the ADR 0030 deliberation:

- **(a) one ADR per tracked field** — 14 new ADRs over the lifetime of [#521](../../issues/521). Audit trail maximally explicit; ledger inflated to ~47 ADRs without proportional decision content. Rejected for the same reason ADR 0030 rejected it: each per-field PR records a per-implementation note, not a per-implementation decision.
- **(b) amend [ADR 0031](0031-dlp-advancedrule-yaml-shape.md) or [ADR 0032](0032-dlp-generic-locations-shape.md) in place** — the natural "amend ADR 0015 in place" analogue for label policies. Rejected because neither 0031 nor 0032 is the rule-shape parent; both are about specific predicate / location wire-shape transforms. The DLP reconciler never had a single rule-shape ADR. Amending the wrong parent would conceal the actual decision provenance.
- **(c) this ADR becomes the rule-shape parent; future per-field PRs amend this ADR in place if the field maps onto a category we already framed here, or ship as their own ADR if a new design question arises.** Hybrid; the option ADR 0030 chose, adapted to the DLP rule's missing-parent situation by promoting *this* ADR to the parent role.

A repo-wide audit of how prior multi-PR rollouts handled the same shape question (2026-06-04):

| Precedent | Shape used | Why |
|---|---|---|
| [ADR 0030](0030-label-policies-tracked-field-expansion.md) (label-policies, 7 fields) | one umbrella ADR; amend 0015 in place for 4 fields; new ADR for 3 fields with new design questions | proved the template; 4 of 7 fields amend in place, 3 of 7 needed new ADRs |
| [ADR 0026](0026-glossary-custom-classifications-reconciler.md) (glossary + classifications, 2 reconcilers) | one ADR, two reconcilers, no per-reconciler ADR | both reconcilers obey the same data-plane drift contract; per-script differences were mechanical not architectural |
| Per-field `advancedSettings:` keys (label-policies) | none added yet; allowlist still at 3 | the prior pattern is "add one per follow-up with a Learn citation"; amend-in-place baseline |

Shape (a) produces 14 ADRs whose decision content is each "we now track field X because Microsoft Learn page Y documents the cmdlet shape and the rule's blast-radius concern is bounded as follows." That is per-implementation, not per-architecture. The ADR ledger would balloon from 32 to 46 without proportional decision content.

Shape (b) is mechanically wrong: neither ADR 0031 nor ADR 0032 is the rule-shape parent. The actual implicit parent is "the rule shape established by [#362](../../issues/362) / PR #383, which deliberately deferred the long tail." That implicit parent has never been written down. Amending 0031 or 0032 to absorb rule-shape decisions outside their stated scope would be the wrong fix.

Shape (c) — promote this ADR to the rule-shape parent, then default to amend-in-place — is the option that survives any reasonable future. It is also the option [#521](../../issues/521)''s acceptance criteria text most naturally maps onto ("ADR 0033 ... document: prioritized field list, per-field PR shape, drift policy, no-batch-larger-than-3-fields rubric"). The cost is that this ADR carries more decision content than ADR 0030 did (because it has to establish the parent it could otherwise inherit), but that is exactly the right cost to pay once.

## Decision

We will:

1. **Promote this ADR to the rule-shape parent.** Future tracked-field PRs amend this ADR in place via dated "Updated: YYYY-MM-DD" notes on the relevant Decision sub-section, citing the implementing PR + Microsoft Learn references + the GA-vs-preview cmdlet shape used. No new ADR file is created for fields that map onto categories framed here.

2. **New ADR only when a design question this ADR did not anticipate is in scope.** The exact triggers that mandate a new ADR (instead of an amendment) — applied as the lab actually reaches each field — are:
   - **Preview-only cmdlet on the only documented surface.** Adopting a preview surface as part of a tracked field is a design decision this ADR explicitly did not make. Wave 0''s [`.github/instructions/powershell.instructions.md`](../../.github/instructions/powershell.instructions.md) GA-over-preview rule already gates this; a new ADR records the exception. Candidates: none currently; `EndpointDlpRestrictions` and `AlertProperties` are both GA per the [`New-DlpComplianceRule` reference](https://learn.microsoft.com/en-us/powershell/module/exchange/new-dlpcompliancerule).
   - **Cross-plane or cross-product impact this ADR did not contemplate.** A tracked field whose authoring surface crosses out of the Microsoft Purview compliance plane into Microsoft 365 Copilot governance, Microsoft Fabric, or Microsoft Entra moves the architecture decision beyond this ADR''s scope. Candidates: `AlertProperties` (Copilot) and `RestrictAccess` (Copilot) — both may warrant a new ADR depending on whether the Copilot-side surface introduces routing that the existing alerting plane doesn''t already cover.
   - **Security blast radius this ADR explicitly defers for cause.** No field in §3 below is in this category today; the boolean defaults (Batch 1) carry near-zero blast radius and the operator-meaningful strings (Batches 2–4) are non-destructive. If a future batch reaches a field whose drift-policy decision is non-obvious (e.g. a field that controls user-facing UX in a regulated workload), that batch ships with its own ADR.

4. **Per-field PR shape contract.** Each tracked-field PR ships with **all** of the following, in one PR, scoped to 1–3 fields:
   - **Schema:** additive extension to [`data-plane/dlp/policies.schema.json`](../../data-plane/dlp/policies.schema.json). No breaking changes; existing YAML without the new field stays valid. Each new field carries an inline description citing [New-DlpComplianceRule](https://learn.microsoft.com/en-us/powershell/module/exchange/new-dlpcompliancerule).
   - **Reconciler:** the new field surfaces in `ConvertTo-DesiredDlpRuleHash`, `ConvertTo-TenantDlpRuleHash`, `Compare-DlpRule`, and `Get-DlpRuleSplat` in [`scripts/Deploy-DLPPolicies.ps1`](../../scripts/Deploy-DLPPolicies.ps1). Bucket-map style for related fields (precedent: the 18-bucket location map from [PR #525](../../pull/525)).
   - **Pester:** new tests in [`tests/scripts/Deploy-DLPPolicies.Tests.ps1`](../../tests/scripts/Deploy-DLPPolicies.Tests.ps1) following the AST-extract pattern, covering at minimum: schema-validation accept, splat emission, no-drift, drift detection. Data-driven (`-ForEach`) when the field set is uniform (precedent: 12 buckets × 2 contexts in [PR #525](../../pull/525); 10 buckets × 6 contexts in [PR #526](../../pull/526)).
   - **YAML re-export:** [`data-plane/dlp/policies.yaml`](../../data-plane/dlp/policies.yaml) re-exported only if the new field surfaces non-null values in the lab tenant. Otherwise no YAML change (precedent: PR #525 and #526 both shipped without YAML changes because no tenant policy used the new fields).
   - **ADR amendment OR new ADR:** per Decision §2 above. If amending this ADR in place, the diff includes the dated update note in the same PR. If a new ADR, the new ADR file is the *only* doc artifact and this ADR gets a one-line "See also" pointer at the end of the relevant Decision §3 subsection.
   - **Acceptance evidence in the PR body:** mirrors the PR #525 / PR #526 template — schema sanity, splat emission per field, `-WhatIf` summary (0 Create / 0 Update / 0 Remove), WhatIf idempotency, lint (0 Warning+), Pester (all green), Microsoft Learn URL HTTP-200 verification, secrets scan, GUID scan, scope evidence.

5. **No batch larger than 3 fields.** Even if multiple adjacent fields look mechanically identical (e.g. the seven notify-family enums), each PR ships ≤3 fields. This obeys [`docs/project-plan.md`](../project-plan.md) §"Delivery cadence — one feature at a time" with the relaxation that a tight 3-field cluster (Batch 1) counts as one feature when the fields are mechanically uniform and the per-field reviewer load is bounded. Exception: a field that triggers a new ADR per Decision §2 always ships solo.

6. **Drift policy: schema-as-source-of-truth, no auto-sync.** The DLP reconciler does not have an auto-drift-sync workflow (unlike the `auto/labels-drift-sync` and `auto/label-policies-drift-sync` branches that exist for sensitivity labels and label policies). Each tracked-field PR inherits [ADR 0029](0029-source-of-truth-direction-policy.md) §"Default direction" — the YAML is the desired state; portal edits that drift from it are flagged on the next `-WhatIf` and overwritten on the next apply. If a future batch needs `portal-wins` direction for a specific field (e.g. an operator-tunable threshold where the portal is the authoring surface), that batch ships with its own ADR amending ADR 0029.

## Consequences

**Easier:**

- **[#521](../../issues/521) becomes implementable.** Each child issue / PR has a fixed scope (1–3 fields), a fixed PR shape (§4), and a fixed "does this need its own ADR" verdict (§2). [`@idea-intake`](../../.github/agents/idea-intake.agent.md) for each child does not have to re-derive the meta-decision.
- **ADR ledger stays calibrated.** Best case (every field amends in place): 0 new ADRs over the lifetime of [#521](../../issues/521). Worst case (Copilot-impacting fields each get their own ADR): 2 new ADRs (`alertProperties`, `restrictAccess`). Either way well below the 14-ADR worst case of Shape (a).
- **This ADR becomes the single source of truth for DLP rule shape.** A reader looking up "why does `Deploy-DLPPolicies.ps1` track only fields X / Y / Z?" finds the question, the answer, and the deferral history in one file.
- **Direction-policy contract ([ADR 0029](0029-source-of-truth-direction-policy.md)) gains coverage as the tracked-field surface expands.** Each tracked-field PR''s `-WhatIf` evidence proves the contract works for the new field; the rollout naturally grows the acceptance corpus without per-field acceptance-script authoring.
- **Implementation order is decision-driven, not probe-order-driven.** Mechanical-defaults-first means the per-field shape contract is exercised on Batch 1''s three booleans before Batch 4''s structured Copilot-impacting fields have to also prove a new architectural pattern.

**Harder:**

- **This ADR grows.** Up to 6 amendment notes over time (one per batch) means this ADR could reach ~12 KB after [#521](../../issues/521) closes — comparable to ADR 0030''s expected end-state but less than ADRs 0028 / 0029. The Update-note pattern handles this readably.
- **The "does this warrant a new ADR" judgment is non-mechanical.** The triggers in Decision §2 are stated explicitly, but the meta-call still requires reviewer judgment when an edge case appears. Mitigation: the [`@idea-intake`](../../.github/agents/idea-intake.agent.md) Step 0 for each child issue restates the trigger list and the verdict before [`@artifact-resolver`](../../.github/agents/artifact-resolver.agent.md) begins.
- **Per-field PR size is non-trivial.** Schema + reconciler + Pester + ADR amendment (or new ADR) + optional YAML re-export + acceptance evidence. Comparable in line count to PR #525 (430 lines) and PR #526 (428 lines) but with a different field shape per batch. Mitigation: the per-field PRs are still single-feature-or-tight-cluster, so reviewer cognitive load stays bounded.
- **No combined refactor opportunity.** A combined "expand all 14 fields in one branch" approach could have factored out a shared `Compare-RuleScalarField` helper. Per-PR cadence means each PR pays the helper-extraction cost individually or duplicates the inline logic. ADR-acknowledged trade-off; per-PR clarity wins, same as ADR 0030 §"No combined refactor opportunity."

**Security principles (from [`.github/instructions/security.instructions.md`](../../.github/instructions/security.instructions.md)):**

- **#4 (least privilege).** Smallest-blast-radius-first ordering means the high-impact `endpointDlpRestrictions` / `alertProperties` / `restrictAccess` fields ship only after the contract is exercised on Batches 1–3''s scalars; the lab''s blast-radius math stays bounded at every step.
- **#9 (idempotent and reversible).** Each tracked-field PR ships its own `-WhatIf` evidence and round-trip Pester coverage; per-PR cadence means a faulty field can be reverted in isolation without disturbing the others. Same property the direction-policy contract relies on.
- **#10 (OWASP-aware).** Each new tracked field is a new operator-visible knob; per-PR review keeps the security-specialist persona''s attention focused on a tight 1–3 field cluster at a time, instead of spread across 14.

**Project-plan items:**

- Does not directly tick any [`docs/project-plan.md`](../project-plan.md) §5 row. [#521](../../issues/521) is cross-cutting expansion of an existing reconciler, not a §5.3 DLP row (that row is the Phase-1 drift-closure work, already separately scoped at [#362](../../issues/362) and ticked).
- Unblocks: the per-batch child issues to be filed against [#521](../../issues/521). [`@idea-intake`](../../.github/agents/idea-intake.agent.md) for each cites this ADR''s Decision §3 ordering and Decision §2 verdict.
- Does not affect: [#65](../../issues/65) (sensitivity labels — own ADR 0015 + ADR 0030 line), [#67](../../issues/67) (auto-labeling — own ADR-0015-like deferral set), Wave 2+ on any other §5 row.

## Alternatives considered

- **(a) One ADR per tracked field.** Rejected per Context — inflates the ledger with per-implementation notes that do not contain per-implementation *decisions*, and dilutes this ADR''s role as the rule-shape parent.
- **(b) Amend [ADR 0031](0031-dlp-advancedrule-yaml-shape.md) or [ADR 0032](0032-dlp-generic-locations-shape.md) in place for every field.** Rejected per Context — neither is the rule-shape parent; both are about specific predicate / location wire-shape transforms. Amending the wrong parent would conceal the actual decision provenance.
- **One omnibus PR that ships all 14 fields at once.** Rejected per Decision §5. The combine-argument wins (one auth call, one workflow run, one Pester sweep) are dwarfed by the loss of per-batch acceptance attribution, per-batch reversibility, and per-batch reviewer focus — exactly the same calculation [ADR 0030](0030-label-policies-tracked-field-expansion.md) §"No combined refactor opportunity" and [ADR 0026](0026-glossary-custom-classifications-reconciler.md) §"Two reconcilers, not one" made.
- **Defer the meta-decision; let each [`@idea-intake`](../../.github/agents/idea-intake.agent.md) child interview re-derive it.** Rejected because the same problem ADR 0030 hit (the [#486](../../issues/486) intake refused to start without a meta-decision) is structurally identical here — the meta-decision is itself an architectural call that blocks per-field work, and re-deriving it across 6+ batches costs more reviewer time than ratifying it once here.
- **Use [`.github/instructions/`](../../.github/instructions/) instead of an ADR**, mirroring how [ADR 0029](0029-source-of-truth-direction-policy.md) pinned the direction-policy *contract* in instructions and let the rollout proceed without per-domain ADRs. Rejected for the same reason ADR 0030 rejected it: the meta-decision being recorded here is about *which artifact records each field-expansion decision*. That is an ADR-shaped question (architecture decision about the documentation artifact), not an instruction-shaped one.
- **Skip the priority list; track all 14 fields in alphabetical order.** Rejected because smallest-blast-radius-first is the property that lets Batch 4''s structured Copilot fields ship after the per-field PR shape contract is already proven on three boolean defaults. Alphabetical order would land `accessScope` (operator-meaningful enum) and `alertProperties` (structured Copilot hashtable) before the trivial boolean defaults — exactly the inversion the ADR 0030 §3 ordering rejected for label policies.
