# 0061 — Custom Sensitive Information Types as verbatim rule-package XML plus a YAML manifest

- **Status:** Accepted <!-- Proposed | Accepted | Superseded by NNNN | Deprecated -->
- **Date:** 2026-07-22
- **Gates:** [#48](../../issues/48) — the fourth per-feature convergence pass (Sensitive Information Types): a new custom-SIT desired-state surface (`data-plane/classifications/sit-rule-packages.yaml` + `data-plane/classifications/rule-packages/*.xml`) and its reconciler `scripts/Deploy-SITRulePackages.ps1`.
- **Deciders:** @marcusjacobson
- **Related:** [ADR 0056](0056-template-ships-empty-desired-state.md) (carve-out 1 keeps `sit-catalog.yaml` populated as reference data — this ADR adds the custom-SIT *desired-state* surface it is the counterpart to), [ADR 0016](0016-auto-label-policy-shape.md) (established that EDM / document-fingerprint conditions are non-representable and out of scope — this ADR extends that exclusion to the SIT-authoring surface), [ADR 0055](0055-identifier-shaped-residual-scan.md) (the GUID-shaped residual scan the repo-authored pack GUIDs must survive), [ADR 0029](0029-source-of-truth-direction-policy.md) / [ADR 0052](0052-destructive-confirmation-gate-at-script-layer.md) (the reconciler contract this script inherits).

## Context

`data-plane/classifications/sit-catalog.yaml` (ADR 0056 carve-out 1) is a **read-only reference catalog** of every Sensitive Information Type (SIT) visible to the tenant — 327 Microsoft built-ins as of [#48](../../issues/48). It exists so downstream label / auto-label / DLP YAML can reference SITs by GUID; it is not desired state, and `scripts/Sync-SITCatalog.ps1` only implements the pull (`-ExportCurrentState`) half.

That script's header deferred an apply path to `New-/Set-/Remove-DlpSensitiveInformationType`. Live verification against Security & Compliance PowerShell (`Get-Command -Syntax`, [#48](../../issues/48) Phase 2.1) confirmed what Microsoft Learn documents but the header missed: **that cmdlet family manages document-fingerprint SITs only** (`-Fingerprints`, `-IsExact`, `-ThresholdConfig`). Custom pattern SITs — the regex/keyword detectors this repo actually wants to manage as code — are created and updated through a **different cmdlet family** that takes a whole **rule-package XML** blob: `New-/Set-/Get-/Remove-DlpSensitiveInformationTypeRulePackage -FileData <byte[]>`.

Phase 2.1's live inspection of the four rule packages on the dev tenant established the design constraints:

1. **GUIDs are author-declared inside the XML.** `<RulePack id>` and each `<Entity id>` are attributes the author writes, not values the service assigns. Importing the same XML into two tenants therefore yields the *same* SIT GUIDs — the mechanism that makes downstream `sitId` references portable across environments exactly like built-ins.
2. **The service stamps `lastModifiedTime` on every `<Entity>` at write time.** A re-export of a pack never byte-matches the XML that was uploaded, because the service rewrites those timestamps (and pack-level `WhenChanged`). Any drift comparator that byte-compares exported-vs-authored XML would report permanent false drift.
3. **`Get-DlpSensitiveInformationTypeRulePackage` returns no `RulePackId` property.** The pack's GUID is recoverable only from the `Identity` distinguished-name tail or by parsing `<RulePack id>` out of the XML.
4. **Every tenant carries Microsoft-managed packs that must never be touched:** the built-in `Microsoft Rule Package` (~2 MB, `Publisher = Microsoft Corporation`), a `Document Fingerprint Rule Package` (identified by the `IsFingerprintRuleCollection = True` flag, under a fixed all-zero-sentinel bucket GUID), any EDM pack (XML namespace `http://schemas.microsoft.com/office/2018/edm`), and `Microsoft.SCCManaged.CustomRulePack` — the container Microsoft auto-provisions in **every** tenant under the fixed GUID `5DA58D7A-25F1-4205-93D0-BEB10054C503` to hold **portal-created** custom SITs. Both the lab and dev tenants already have real, portal-authored custom SITs sitting in their `5DA58D7A` container (lab: "ABA Routing Number copy"; dev: a hand-authored "Retail Loyalty ID" plus a "U.S. ITIN copy"). The shared GUID across tenants reflects the shared *container*, not shared content — and its `<Publisher id>` is the tenant's own real Entra tenant GUID.
5. **A rule-package update is silently ignored unless the XML `<Version>` is bumped** (Microsoft Learn; corroborated by the `Version` property the pack object exposes). There is a hard limit of 10 rule packages per tenant and an effective ~770 KB upload ceiling.

The lab targets a single tenant and a single human per the "Environment and identifier boundaries" section of [`.github/copilot-instructions.md`](../../.github/copilot-instructions.md), but the same code path must stay safe against a shared tenant and must never mutate a Microsoft-managed pack. That is what constrains the shape below.

## Decision

We will manage custom pattern SITs as **repo-authored rule packages**, each stored as a **verbatim rule-package XML file** and indexed by a YAML **manifest**, reconciled by a new `scripts/Deploy-SITRulePackages.ps1` that conforms to the standard full-circle reconciler contract. Specifically:

1. **Desired-state shape — verbatim XML + manifest, not a YAML projection.**
   - Each rule package is a byte-faithful `.xml` file under `data-plane/classifications/rule-packages/`.
   - `data-plane/classifications/sit-rule-packages.yaml` is the manifest: a `rulePackages:` list, each entry `{ name, rulePackId, version, file, sits: [{ name, id }] }`, where `rulePackId` / `version` / the entity `id`s **must** match the values declared inside the referenced XML (the reconciler validates this cross-reference and emits a `Blocked` row on mismatch).
   - The manifest ships **empty** (`rulePackages: []`) on `main` per [ADR 0056](0056-template-ships-empty-desired-state.md), is populated only on the `lab` / `dev` operator branches, and gets a worked, non-deployed example under `examples/data-plane/classifications/`.
   - We reject the alternative of authoring SITs as declarative YAML compiled to XML at apply time: a YAML→XML compiler is exactly the export/apply projection-bug surface that already cost the auto-label pass a round trip (the `workload` pipe-join, [#20](../../issues/20)). Verbatim XML has no projection layer to get wrong and is GUID-stable by construction.

2. **GUID authoring and identifier boundaries — `<RulePack id>` and `<Entity id>` are real random GUIDs; `<Publisher id>` stays in the reserved synthetic namespace.**
   - **Resolved (Phase 3, 2026-07-22): the service REJECTS the synthetic namespace for `<RulePack id>`.** A live `New-DlpSensitiveInformationTypeRulePackage` call against the dev tenant with `<RulePack id="00000000-0000-0000-0000-000000000071">` failed validation: `"00000000-0000-0000-0000-000000000071" is an invalid identifier for custom classification rule collection. Classification rule collection identifiers which begin with "00000000" are reserved for out-of-box classification rule collections.` This falsifies the premise this decision originally rested on. The documented fallback (below) is what shipped.
   - **Shipped shape.** `<RulePack id>` and every `<Entity id>` (and its `<LocalizedStrings>/<Resource idRef>` twin) are ordinary random GUIDs (`[guid]::NewGuid()`), author-declared exactly as before — importing the same file into two tenants still yields the same SIT + rule-pack GUIDs, the cross-tenant join key is unchanged. `<Publisher id>` alone stays in the reserved synthetic namespace `00000000-0000-0000-0000-<counter>`: the live Apply that rejected a synthetic `RulePack id` **accepted** a synthetic `Publisher id` unchanged, so no fallback was needed there, and keeping it synthetic is what keeps committed XML free of any tenant-real identifier (decision unchanged from the paragraph below).
   - **No XML-aware claim mode was needed.** The anticipated fallback ("extending the `catalogKeys` carve-out to `rule-packages/*.xml`, which would require adding an XML-aware claim mode to the [ADR 0055](0055-identifier-shaped-residual-scan.md) scan") turned out to be unnecessary: the scan's `catalogKeys` rule acquits a GUID **by value**, independent of which file or syntax it appears in. Because `sit-rule-packages.yaml` was already a `catalogKeys` file (`rulePackId`, `sits[].id`) as future-proofing, and the manifest's `rulePackId` / `sits[].id` values are required to equal the XML's `<RulePack id>` / `<Entity id>` (the reconciler's own cross-reference check), the same GUID value appearing again inside the `.xml` file is acquitted for free once it is acquitted in the YAML. Verified clean: `Test-IdentifierResidue.ps1` reports 0 unclaimed identifiers over the pilot's XML + manifest.
   - A `catalogKeys` entry for `sit-rule-packages.yaml` (`rulePackId`, `sits[].id`) is still added to `.github/agents/tenant-placeholders.yaml` as future-proofing (it contributes nothing while the manifest ships empty, mirroring the `dlp/policies.yaml` precedent), so a spin-off that ever populates real-random GUIDs has a reviewed home for them.
   - The `<Publisher id>` is the one field the portal populates with the live tenant GUID (observed on dev's `5DA58D7A` pack); pinning it to a synthetic-namespace constant is what keeps committed XML free of any tenant-real identifier.

3. **Manage/prune scope is exactly the manifest; Microsoft-managed packs are permanently out of scope.**
   - The reconciler only ever creates, updates, or prunes rule packages whose `rulePackId` appears in `sit-rule-packages.yaml`. A tenant pack not in the manifest is never touched (no prune), and a manifest entry absent from the tenant is a `Create`.
   - Four identifier classes are a **reserved denylist**: the built-in `Microsoft Rule Package`, any `IsFingerprintRuleCollection` pack (the fixed all-zero-sentinel bucket), any EDM pack (namespace `.../office/2018/edm`), and `Microsoft.SCCManaged.CustomRulePack` (`5DA58D7A-25F1-4205-93D0-BEB10054C503`). If any appears in the manifest, or would be matched for prune, the reconciler emits a `Blocked` row and refuses — defense in depth so a repo pack can never collide with, adopt, or delete a Microsoft-managed container. Portal-created custom SITs stay portal-owned.

4. **Drift is compared canonically, never byte-for-byte.** The `NoChange` decision normalizes both the desired XML and the tenant's exported XML into a canonical form — comparing `<RulePack id>` + `<Version>` + each `<Entity>` (by `id`, pattern structure, `recommendedConfidence`) + `<Regex>` / `<Keyword>` / `<LocalizedStrings>` — and **ignores service-stamped timestamps** (`lastModifiedTime`, `WhenChanged`, `LastModifiedTime`). Byte-compare is rejected on the Phase 2.1 evidence that the service rewrites `lastModifiedTime` on every write. Phase 3's round-trip gate on dev validated the comparator end-to-end (`-Apply` then `-WhatIf` reported all-`NoChange` only after one additional fix): the service also stamps a **default `relaxProximity="false"`** onto every `<Entity>` at write time when the authored XML omits it — undocumented on Microsoft Learn as of 2026-07-22, discovered only by the live round-trip. The comparator now **default-fills** `relaxProximity` to `"false"` when absent (rather than stripping it as volatile), so an explicit non-default `relaxProximity="true"` in authored XML remains real, comparable drift while an untouched pack still reports `NoChange` after export.

   This is the second time a live round-trip found a service behavior the design phase could not: the first (immediately below) is `Get-DlpSensitiveInformationTypeRulePackage`'s `.Identity` returning an **empty string** for the built-in `Microsoft Rule Package` specifically (it has no per-tenant Configuration object) — a case Phase 2.1's live inspection did not happen to hit, which crashed `Get-RulePackIdFromIdentityDn`'s mandatory-string parameter binding on every plan run, in every tenant, before any Create could even be planned. Fixed by allowing (and short-circuiting on) an empty/null `IdentityDn`: the pack is still correctly classified as reserved via its `RuleCollectionName`, which `Test-ReservedRulePack` checks first and which never depends on `RulePackId`.

5. **A content change requires a `<Version>` bump.** Because the service silently ignores an updated pack whose `<Version>` is unchanged, the reconciler treats "canonical content differs but `<Version>` is unchanged" as a `Blocked` row (not a silent `Update`), forcing the author to bump the version deliberately. Pre-flight `Blocked` rows likewise cover the tenant limits: more than 10 total packages, or an upload exceeding ~770 KB.

6. **EDM and document-fingerprint SITs remain out of scope**, consistent with [ADR 0016](0016-auto-label-policy-shape.md). EDM's data-store upload (`EdmUploadAgent.exe`) requires an interactive user in `EDM_DataUploaders` and is not app-only scriptable; document-fingerprint SITs are name-keyed (no author-declarable GUID) and tenant-local. `Sync-SITCatalog.ps1` still *records* any such SITs in the read-only catalog when present, but `Deploy-SITRulePackages.ps1` does not manage them.

7. **`sit-catalog.yaml` stays exporter-owned reference data.** Custom SITs deployed by this reconciler enter the catalog only via a later `Sync-SITCatalog.ps1 -ExportCurrentState`, never by hand. This ADR therefore also **supersedes the stale apply-path note** in `Sync-SITCatalog.ps1`'s header (which pointed at the fingerprint-only `*-DlpSensitiveInformationType` cmdlets); the header is updated to point here.
   - **Caveat found in Phase 3: a raw `-ExportCurrentState` overwrite is not safe to commit verbatim on a tenant carrying un-managed discoveries.** `Sync-SITCatalog.ps1` redacts a publisher only when it is GUID-shaped (to the zero GUID); a publisher *name* is kept verbatim by design (decision intent: tenant-custom publisher org names are expected content). On the dev tenant this is a problem for the eleven pre-existing, un-managed SITs [#48](../../issues/48) Phase 1 already found and deliberately excluded (decision 6): eight document-fingerprint SITs plus one EDM SIT report the tenant's own `*.onmicrosoft.com` domain as `publisher`, a real tenant identifier, in a PUBLIC template file. So the Phase 3 catalog refresh was **hand-filtered** to the one new repo-managed row rather than committing the exporter's raw output. `tests/data-plane/ShippedDesiredState.Tests.ps1` carve-out 1's content assertion now enforces this mechanically: a non-`Microsoft Corporation` catalog entry is allowed only if its `id` is declared in `sit-rule-packages.yaml` (i.e., repo-managed); anything else — an un-managed discovery or any other real tenant data — fails the test. A future export that adopts more repo-managed customs is safe by construction; a future export on a tenant carrying new un-managed discoveries needs the same hand-filtering, not a blind `-Force` overwrite.

## Consequences

**Easier / unblocked:**

- [#48](../../issues/48) can ship a conforming reconciler with a byte-faithful, GUID-stable desired state, and a pilot custom SIT can be proven to deploy with identical GUIDs on both tenants (Phase 3 → Phase 4).
- Downstream label / auto-label / DLP YAML can reference a repo-managed custom SIT by the same GUID on every environment, exactly as they already reference built-ins.
- The drift comparator is defined on evidence, not on a guess about whether the service round-trips bytes — the single most common way these reconcilers ship a permanent-false-drift bug (the [#20](../../issues/20) class) is designed out.

**Harder / accepted:**

- Authoring a custom SIT means writing rule-package XML by hand (or exporting and editing), not filling in a friendly YAML. This is the deliberate trade for byte-fidelity; the manifest + schema + a worked example under `examples/` mitigate it, and custom SITs are expected to be few.
- Portal-created custom SITs already in each tenant's `Microsoft.SCCManaged.CustomRulePack` are **not** brought under management by this decision — they remain portal-owned and unmanaged. Adopting them would require co-managing a Microsoft-provisioned container, which point 3 explicitly refuses. If they ever need to be as-code, they must be re-authored as a repo pack with repo GUIDs and the portal originals retired by hand — a separate, deliberate follow-up, not silent adoption.
- `tests/data-plane/ShippedDesiredState.Tests.ps1` carve-out 1 needed updating twice, in two different PRs, both deliberate and reviewed: the writer-script assertion (Phase 2, this reconciler's PR) widened its unanchored `(New|Set|Remove)-DlpSensitiveInformationType` match — which already matched the new `*RulePackage` cmdlet names as a substring — to sanction `Deploy-SITRulePackages.ps1` as a second catalog-adjacent writer; the content assertion (Phase 3, the first real post-deploy export) changed from "every entry is `publisher: Microsoft Corporation`" to "every non-Microsoft entry's `id` is declared in `sit-rule-packages.yaml`" once a real repo-managed custom SIT existed to prove it against (point 7's caveat).

**Security posture.** No new stored secret or identity. The reconciler authenticates via the existing app-only local-cert / Key-Vault path ([ADR 0028](0028-co-equal-local-cert-credential.md) / [ADR 0011](0011-certificate-lifecycle.md)). The one tenant-real identifier the portal puts in pack XML (`<Publisher id>`) is pinned to a synthetic constant for repo-authored packs, so committed XML upholds the [ADR 0055](0055-identifier-shaped-residual-scan.md) no-real-identifiers rule. The reserved denylist (point 3) upholds least-surprise: the reconciler cannot delete or overwrite a Microsoft-managed pack even if a manifest is mis-authored.

## Alternatives considered

1. **Do nothing / keep `Sync-SITCatalog.ps1` export-only.** Reject. Custom SITs then cannot be represented as code at all, and the script's header keeps pointing at an apply path (the fingerprint-only cmdlets) that does not do what it claims — a standing correctness trap for the next author.
2. **Declarative YAML compiled to XML at apply time.** Reject as the default. It is more idiomatic with the repo's ~20 pure-YAML reconcilers and friendlier to author, but it reintroduces the export/apply projection-bug class ([#20](../../issues/20)): the compiler must round-trip every XML feature (patterns, proximity, confidence tiers, keyword dictionaries, localized strings) losslessly, and any gap is a silent deploy bug. Revisit only if custom SITs grow numerous enough that hand-authored XML becomes the bottleneck.
3. **Adopt and manage the existing `Microsoft.SCCManaged.CustomRulePack` (`5DA58D7A`) as the repo's pack.** Reject. It is a Microsoft-provisioned, portal-managed container with a fixed cross-tenant GUID and the live tenant ID as its publisher; co-managing it as code means fighting the portal for ownership of the same object and risking deletion of portal-authored SITs on every prune. A repo pack with its own GUID and synthetic publisher is cleanly separable.
4. **Manage custom SITs one `<Entity>` at a time rather than whole packs.** Reject. The cmdlet surface has no per-entity create/update for pattern SITs — the unit of write is the whole rule package (`-FileData`), and a partial write replaces the package. Whole-pack management matches the API's actual granularity.
