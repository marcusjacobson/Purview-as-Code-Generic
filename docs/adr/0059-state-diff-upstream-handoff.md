# 0059 — State-diff upstream hand-off and the operator promotion chain

- **Deciders:** @marcusjacobson

## Context

The operator of this template runs a private downstream repository (ADR 0057 §8: `main` mirroring this repo fast-forward-only, `dev` and `lab` operator branches carrying tenant-tailored, populated state). Fixes discovered while operating those branches often generalize and belong back in this template. The first design for that outbound hand-off was **PR-centric**: an `upstream-candidate` label applied by a merge-time interview marked each downstream PR owed upstream, a three-agent chain (scout / porter / tracker) cherry-picked one source commit at a time oldest-first into a clone of this repo, and a ledger recorded completed ports keyed on source merge-commit SHAs.

That design had structural problems the owner objected to directly ("we made the effort to move away from the PR to prefer the checksum approach"): discovery rode on **mutable GitHub label state** and on a human answering an interview correctly at every merge; unlabeled or pre-process PRs were invisible; oldest-first cherry-picking created ordering discipline and conflict risk that existed only to serve the porting mechanism; and the ledger could drift from reality because "done" was *recorded*, not *observed*.

Two measured facts made a content-based redesign obviously right. First, because the operator `main` is a byte-faithful fast-forward mirror of this template, "what does the operator carry that the template lacks" is answerable **inside one repository** as a pure content diff — no clone, no API call, no label. Second, the operator repo's entire committed `main`↔`lab` divergence was exactly its tenant/operator surfaces (15 files at measurement time, every one enumerable in advance), so the diff's noise floor after declaring those surfaces is zero.

Separately, the operator's branch documentation described `lab` and `dev` as "peers, never merged into each other," which contradicted the owner's intended flow: changes prove on the dev tenant first (dev credentials), then promote to the lab tenant (lab credentials), and only then feed upstream.

## Decision

1. **Discovery is a state diff, not a queue.** Owed upstream `=` `git diff main lab` minus the operator surfaces declared in a machine-readable scope manifest (`.github/agents/upstream-scope.yaml`). Classification is first-match-wins: `portableExceptions` (template content living inside an operator glob, e.g. fixtures under `data-plane/**`), then `operatorSurfaces` (tenant-tailored files per the ADR 0046 manifest, per-environment files per ADR 0057 §4, populated desired state per ADR 0056, derived artifacts, each repo's own changelog, and the hand-off machinery itself), and everything unmatched is **portable — owed by definition**. The genericness judgment ("does this belong upstream?") is not hash-derivable; it survives as the owner's confirmation of each port set, and a path ruled operator-local is recorded in the manifest **once** and never asked about again. Tenant-tailored files that also receive template updates are marked `mixed`: they are suppressed from the broad diff, and the scan separately lists commits touching them since the last recorded run (the skim signal), with porting from them a deliberate hunk-level act.

2. **Detection is automatic and issue-only.** A weekly scheduled workflow (`upstream-delta-watch.yml`, ADR 0044 pattern: `contents: read` + `issues: write`, no OIDC, no tenant access) runs the scan and opens one deduplicated `upstream-delta` issue when anything is owed or the mirror is stale against this repo as last fetched; a quiet week stays quiet. The same scan runs on demand. The scan **fails fast** when `main` is not an ancestor of the port branch, and warns STALE when local `main` no longer matches `upstream/main` — porting against a stale mirror is the one race in the model, and the guard's answer is always: sync down first, re-scan, re-port.

3. **Porting is a conflict-free state copy, batched.** The port happens in the separate push-capable clone (ADR 0045: the operator repo's own `upstream` remote stays fetch-only): fetch the operator `lab` across a local-path remote, `git restore --source` the confirmed files (plus `git rm` for deletions) onto a `sync/ops-delta-<checksum>` branch cut from this repo's current `main`, run `Test-IdentifierResidue.ps1` **in the clone** with no path excludes (ADR 0055 — any finding is a hard stop), add a template-side changelog entry, and open **one batch PR** carrying the file table, each file's source commits (`git log main..lab -- <path>` — attribution without cherry-picking), and the delta checksum. The merge-method gate of ADR 0057 §9 applies at open time (`merge-commit` label when the batch touches workflow or multi-environment contract surfaces, erring toward applying it). Cherry-picks, oldest-first ordering, and one-fix-per-PR are retired — they existed to serve a porting mechanism that no longer exists.

4. **The ledger is a per-run checksum journal, not the source of truth.** Each recorded scan writes one row — base/head ref SHAs, file count, a SHA-256 delta checksum over the sorted owed set, template PR, `Open`/`Ported` — keyed idempotently on the checksum, dual-written to `lab` and `dev` by direct commit (the standing narrow exception to "always a PR", scoped to this one file). Any historical scan state is reconstructable from the recorded SHAs, and every prior ledger state stays reachable via `git log -p`. **Closure is observed, never recorded:** after the template PR merges and the sync-down runs, the ported files are byte-identical on both sides and the diff collapses to empty on its own — the next scan reading "owed: nothing" is the proof, so no bookkeeping can drift from reality.

5. **The operator branch model is a promotion chain, with the template as the routing point between `lab` and `main`.** `dev` is intake: every change lands there first and deploys to the dev tenant under the `dev` Environment's credentials (ADR 0057 §1 routing). `lab` takes validated changes by ordinary merge and re-validates under the lab tenant's credentials. `lab` **never merges directly into `main`** — a git merge would carry tenant tailoring, break the ADR 0056 emptiness guard on `main`, and destroy the mirror identity the state diff depends on. Instead `lab`'s tenant-neutral delta ports upstream (this ADR's mechanism) and returns to `main` as a fast-forward, after which the sync-down merges `main` into both operator branches (an ordinary merge commit each — fast-forward is impossible on branches carrying operator commits by design). An emergency fix landed directly on `lab` is followed by merging `lab` back into `dev` to restore one-directional convergence.

6. **Retired outright:** the `upstream-candidate` / `upstream-ported` labels, the merge-time "belongs upstream?" interview, and the scout / porter / tracker agent chain — replaced by one `@upstream-handoff` agent running scan → confirm → record → port → PR → close-out.

## Consequences

**Easier.** Discovery is mechanical and cannot miss unlabeled work — content that differs cannot hide the way an unlabeled PR could, and a local-only hot-fix surfaces in the very next scan and weekly issue. Ports cannot conflict, so batching is free and ordering discipline disappears. The genericness question is asked once per path instead of once per merge. "Did we forget anything" resolves to a command, not a recollection.

**Harder / accepted trade-offs.** The scope manifest must stay faithful to the surfaces it declares (a new tenant surface added to the ADR 0046 manifest must be mirrored there; the mixed-surface skim list is the safety net for generic fixes buried in tailored files). Batch PRs trade per-fix template PR granularity for throughput — the per-file source-commit lists in the PR body preserve attribution. The port has one race — this repo moving between the operator's last fetch and the port — which the currency guard converts into "sync down and re-scan": **observed live on the very first production port**, where two files of a prepared batch were superseded mid-flight by upstream commits; the guard's abort-resync-rescan path produced a correct second batch (template PR #142) and left the superseded scan row in the ledger as an honest `Open` record.

**Security posture.** Unchanged where it matters: the ADR 0055 identifier gate runs against every ported state in the clone with no exclusions; ADR 0056's emptiness guard keeps enforcing on `main`; the watch loop holds no tenant credentials and can only read contents and write issues; ADR 0045's no-push-back boundary is untouched (the operator repo still cannot push here — only the separate clone can, and only via PR).

## Alternatives considered

**A — High-water-mark commit tracking** (ledger stores the last-ported SHA per branch; candidates = `git log <mark>..HEAD` filtered to neutral paths). Reject: per-branch marks to maintain across two operator branches, tenant-tailoring commits interleaved with generic fixes make the filter the hard part, ordering returns, and a mark can silently desync from reality — the exact failure class the redesign exists to remove.

**B — Hybrid: state-diff discovery, cherry-pick execution** (find the delta by content, then port the commits that produced it). Reject: reintroduces cherry-pick conflicts and ordering for no gain — the only thing commits add over state is attribution, which the PR body's per-file source-commit lists preserve for free.

**C — Status quo (label queue + merge-time interview + three-agent chain).** Reject: the owner's objection stands; discovery depends on mutable label state and a human answer at every merge; unlabeled history is invisible; per-PR ceremony scales with merge volume rather than with what is actually owed.
