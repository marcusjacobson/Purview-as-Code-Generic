## Delivery cadence — one item at a time

> [!NOTE]
> **v1 cadence concluded on 2026-05-25** with the merge of PR #343 (Wave 4b-ii). The rules below remain as historical reference and as the operating cadence for any future v2 work; no v1 items remain in flight.

Work proceeds strictly item-by-item down the project-board queue. No batching, no parallel waves. Per [ADR 0014](../adr/0014-agents-as-default-entry-point.md), the default flow is agent-led:

1. [`@idea-intake`](../../.github/agents/idea-intake.agent.md) — classify the work, enforce the §4 dependency-matrix and §8 ADR gates inline when the work maps onto a Progress-checklist row, draft the issue, create and push the scoped feature branch on confirmation.
2. [`/build-item`](../../.github/prompts/build-item.prompt.md) — implement, locally validate, deploy to the lab, verify exit criteria. Loops until the author replies `ready for checkin`. **Never commits or pushes.** Shared validation engine for both human-driven and agent-driven flows.
3. [`@artifact-resolver`](../../.github/agents/artifact-resolver.agent.md) — run the `/build-item` validation, stage scoped paths, run secrets-scan, commit with Conventional Commits, push, open the PR with all required sections.
4. [`@owner-approval`](../../.github/agents/owner-approval.agent.md) — apply `owner-approved`, wait for the [`pr-auto-merge.yml`](../../.github/workflows/pr-auto-merge.yml) workflow to squash-merge, run local cleanup, confirm the project-plan checkbox tick, prompt for the next item.

[`@squad`](../../.github/agents/squad.agent.md) is reserved for content-creation interviews ([`/add-classification`](../../.github/prompts/add-classification.prompt.md), [`/add-data-source`](../../.github/prompts/add-data-source.prompt.md)) and persona-led discussion. It is not a lifecycle agent.

Rules that make this cadence work:

- **One item per PR.** If a diff starts growing beyond the scoped item, stop and split.
- **Dependencies are hard gates.** An item cannot start until every item it depends on in §4 is ticked. If a blocker is discovered mid-item, abandon the PR and open the prerequisite item first. `@idea-intake` Step 0 enforces this for Progress-checklist items.
- **Re-runnable.** Every deploy must be idempotent.
- **No skipping.** Triviality is not an excuse to batch.
- **ADRs block the first item they gate, not the whole wave.** An open §8 question blocks the Progress checklist item(s) named in its row, not every item in the same wave. Resolved historical example: Q7 ([ADR 0024](../adr/0024-unified-catalog-folder-placement.md)) gated only Wave 4b ([#83](../../../issues/83)), never Wave 4a.
- **Apply-path-hardening follow-ups gate the next wave.** A new wave does not start until the prior wave's apply-path-hardening follow-up issue, if any, is merged. This prevents the off-plan remediation queue from outrunning the project plan. Most recent chain: [#267](../../../issues/267) (Wave 2d IRM, merged 2026-05-17 as PR #288) and [#271](../../../issues/271) (Wave 2e Communication Compliance, closed 2026-05-17 as superseded by [ADR 0019](../adr/0019-cc-graph-pivot.md)) — both closed, so Wave 3a ([#74](../../../issues/74)) is unblocked on this gate. No current chain remains open; no further waves are planned under v1.

## 1. Scope and constraints

- **Target environment.** Single lab environment: `purview-contoso-lab` in `rg-purview-lab`, region `eastus`, tenant `contoso.onmicrosoft.com`. Any multi-environment story is a future design PR, not part of this plan. See the "Environment and identifier boundaries" section of [`.github/copilot-instructions.md`](../../.github/copilot-instructions.md).
- **Two planes stay separate.** Control plane stays in [`infra/`](../../infra/); data plane stays in [`data-plane/`](../../data-plane/) + [`scripts/`](../../scripts/). Any new top-level folder requires an ADR.
- **Licensing assumption.** M365 E5 or E5 Compliance add-on across the tenant. Waves 1–3 depend on this (IP, M365 governance, DSPM); any gap is a follow-up ADR + issue.
- **Automation identity.** App-only auth via certificate-in-Key-Vault and federated OIDC from GitHub Actions, per [`.github/instructions/security.instructions.md`](../../.github/instructions/security.instructions.md) rules #2 and #3. No stored client secrets.
- **Reference repo is read-only.** [`Azure-Deployment-Pipelines/Purview/`](../../../contoso.onmicrosoft.com/Azure-Deployment-Pipelines/Purview/README.md) is a **pattern source**, not a dependency. The full pattern table lives in [`architecture.md`](../architecture.md#reference-patterns-adopted-from-azure-deployment-pipelines).

## 2. Guiding principles

1. **Dependency-driven ordering.** A solution only ships after everything it reads from ships. This is why Information Protection is Wave 1 (labels are referenced by DLP, auto-labeling, IRM, and DSPM) and Azure governance is Wave 4 (Data Map and Unified Catalog land after the M365 posture surface that does not depend on them — see [ADR 0020](../adr/0020-dspm-before-azure-gov.md)).
2. **Governance-first.** RBAC, Administrative Units (AUs), and the unified audit log land before any solution that depends on them. See [Microsoft Purview roles and groups](https://learn.microsoft.com/en-us/purview/microsoft-365-compliance-center-permissions) and [Turn auditing on or off](https://learn.microsoft.com/en-us/purview/audit-log-enable-disable).
3. **Simulation-first.** Every state-changing solution ships with a dry-run mode before a production mode. Naming follows the reference repo's vocabulary: `TestWithoutNotifications` → `TestWithNotifications` → `Enable`. In this repo, that manifests as `-WhatIf` for control plane and an explicit `mode:` field in data-plane YAML where supported.
4. **Microsoft Learn is authoritative.** Every resource type, cmdlet, REST endpoint, and role name cites a current Learn page in the same PR that introduces it. No training-data-only claims. See [`.github/copilot-instructions.md`](../../.github/copilot-instructions.md) "Grounding" section.
5. **Idempotent and reversible.** Every `Deploy-*.ps1` is safe to re-run. Destructive flags (`-PruneMissing`, `-Force`) are opt-in and carry the `destructive` PR label.
6. **No real data, no real identifiers.** Synthetic substitutes only, per [`.github/instructions/sample-data.instructions.md`](../../.github/instructions/sample-data.instructions.md).

## 3. Solution universe and wave ordering

Per the directive to lead with Information Protection, then M365 governance, then DSPM, then Azure governance (DSPM and Azure governance swapped per [ADR 0020](../adr/0020-dspm-before-azure-gov.md)):

| Wave | Bucket | Solutions | Rationale |
|---|---|---|---|
| 0 | Foundation | Governance (RBAC, AUs, audit log, automation identity) | Hard prerequisite for every wave below. |
| 1 | Information Protection | Sensitivity label taxonomy, label policies, auto-labeling policies | Labels are referenced by DLP, IRM, DSPM, Unified Catalog. Must land first. |
| 2 | M365 governance | Audit → DLP → Data Lifecycle Management + Records → Insider Risk Management → Communication Compliance | Ordered by intra-wave dependencies (see §4). eDiscovery removed per [ADR 0018](../adr/0018-ediscovery-scope.md). |
| 3 | DSPM | Data Security Posture Management, DSPM for AI | Consumes M365 signals from Waves 0–2 (labels, SITs, DLP, IRM, audit). No Data Map dependency on the standard surface ([ADR 0020](../adr/0020-dspm-before-azure-gov.md)). |
| 4 | Azure governance | Purview Data Map (already scaffolded) → Unified Catalog | Deferred behind M365 and DSPM because the lab's higher-value use cases are M365-side and Data Map population requires synthetic Azure data sources. |

Solutions **not** in scope for this plan: Compliance Manager, Information Barriers, Priva, Data Sharing, Microsoft 365 Backup, eDiscovery (see [ADR 0018](../adr/0018-ediscovery-scope.md)). The first five would each require a separate ADR if reintroduced; eDiscovery already has one.

## 5. Out of scope

The following are explicitly **not** part of this plan and would each require a separate design PR:

- Multi-environment deployment (dev / test / prod).
- Customer-managed keys (CMK) for the Purview account.
- Private endpoints for the Purview account (currently `publicNetworkAccess: 'Enabled'` in [`infra/main.bicepparam`](../../infra/main.bicepparam) with a lab-only justification; flipping this is a dedicated PR).
- Compliance Manager, Information Barriers, Priva, Data Sharing, Microsoft 365 Backup.
- Porting the reference repo's Azure DevOps pipelines verbatim. This repo stays on GitHub Actions.
- Windows PowerShell 5.1 compatibility. This repo is pwsh 7.4+ only.
- eDiscovery-as-code (cases, custodians, holds, searches, review sets, exports) — see [ADR 0018](../adr/0018-ediscovery-scope.md). Reusable hold / search query templates are explicitly left open as a future, narrower question.
